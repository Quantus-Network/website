## Introduction

The advent of scalable quantum computers poses an existential threat to every blockchain that relies on elliptic-curve cryptography. ECDSA, EdDSA, and BLS signatures — the foundations of Bitcoin, Ethereum, and most modern chains — can be broken by a sufficiently powerful quantum adversary running Shor's algorithm.

Quantus Network was designed from the ground up to be **quantum-resistant from day one**, not as a retrofit. This whitepaper describes the architecture, cryptographic primitives, consensus mechanism, and economic model that together form a secure, performant, and developer-friendly Layer 1 blockchain.

<Callout variant="info">
Quantus uses CRYSTALS-Dilithium and CRYSTALS-Kyber, both standardized by NIST in 2024 as the primary post-quantum signature and key-encapsulation schemes.
</Callout>

## Problem Statement

### The Quantum Threat

Current estimates suggest that a cryptographically relevant quantum computer (CRQC) capable of breaking 256-bit elliptic-curve keys could arrive between 2029 and 2035. Once such a machine exists:

- **Harvest-now, decrypt-later attacks** become viable — adversaries can record encrypted traffic today and decrypt it once quantum hardware matures.
- **Signature forgery** enables anyone to spend funds from any address whose public key has been revealed on-chain.
- **Chain reorganization** becomes trivial when block proposers' keys can be forged.

### Limitations of Existing Solutions

| Approach | Limitation |
|---|---|
| Post-quantum bolt-on to existing L1 | Requires hard fork; legacy addresses remain vulnerable |
| Hybrid classical + PQ signatures | Doubles signature size; complex upgrade path |
| Hash-based signatures (XMSS, LMS) | Stateful key management; limited number of signatures per key |

Quantus addresses these limitations by integrating post-quantum cryptography into every layer of the protocol from genesis.

## Architecture Overview

The Quantus Network architecture consists of four primary layers:

1. **Cryptographic Layer** — CRYSTALS-Dilithium signatures, CRYSTALS-Kyber key exchange, and BLAKE3 hashing
2. **Consensus Layer** — A DAG-based consensus protocol with instant finality
3. **Execution Layer** — A lightweight virtual machine optimized for financial primitives
4. **Application Layer** — SDKs, wallet infrastructure, and developer tooling

<Callout variant="tip">
Each layer is designed to be independently upgradeable, allowing the network to adopt stronger cryptographic primitives as the field evolves without requiring a full protocol overhaul.
</Callout>

### Cryptographic Primitives

#### CRYSTALS-Dilithium (Digital Signatures)

Dilithium is a lattice-based digital signature scheme selected by NIST for standardization (FIPS 204). Quantus uses Dilithium-3 (NIST Security Level 3), which provides:

- **Public key size:** 1,952 bytes
- **Signature size:** 3,293 bytes
- **Security:** 192-bit classical / quantum security

#### CRYSTALS-Kyber (Key Encapsulation)

Kyber is used for encrypted peer-to-peer communication between nodes. Quantus deploys Kyber-768 (NIST Security Level 3):

- **Public key size:** 1,184 bytes
- **Ciphertext size:** 1,088 bytes
- **Shared secret:** 256 bits

#### BLAKE3 (Hashing)

All Merkle trees, block hashes, and address derivations use BLAKE3 for its:

- Resistance to length-extension attacks
- Hardware-friendly parallelism
- 256-bit output with quantum-safe collision resistance

## Consensus Mechanism

Quantus employs a **Directed Acyclic Graph (DAG)** consensus model combined with a proof-of-stake validator set. Unlike traditional blockchain consensus where blocks form a single chain, Quantus validators emit vertices that reference multiple parent vertices, forming a DAG structure.

### Key Properties

- **Instant finality:** Transactions achieve finality within 1–2 seconds
- **High throughput:** The DAG structure allows parallel transaction processing, supporting 10,000+ TPS in benchmarks
- **Energy efficient:** Proof-of-stake eliminates the energy waste of proof-of-work
- **Byzantine fault tolerance:** The network tolerates up to ⅓ malicious validators

### Validator Selection

Validators are selected through a stake-weighted random sampling process:

1. Validators lock QNT tokens as stake
2. A VRF (Verifiable Random Function) selects the active validator set for each epoch
3. Selected validators propose vertices and participate in consensus voting
4. Rewards are distributed proportionally to stake and uptime

## Reversible Transactions

One of Quantus's most innovative features is its **reversible transaction model**. Within a configurable grace period (default: 60 seconds), a sender can reverse a transaction. This dramatically reduces losses from:

- Accidental sends to wrong addresses
- Transactions initiated under duress
- Smart contract bugs caught immediately after execution

<Callout variant="warning">
Reversibility is optional and configurable per transaction. Merchants and exchanges can accept irreversible transactions for instant settlement when the use case demands it.
</Callout>

### How Reversibility Works

1. Sender submits a transaction with `reversible: true` and a `gracePeriod` parameter
2. The transaction enters a "pending-final" state on the DAG
3. During the grace period, only the sender can issue a `reverse` instruction
4. After the grace period expires, the transaction becomes irreversible and is treated identically to a standard finalized transaction

## Zero-Knowledge Proofs

Quantus integrates zero-knowledge proofs (ZKPs) at the protocol level to enable:

- **Private transactions** — Shielded transfers where amounts and participants are hidden
- **Proof aggregation** — Batch verification of many transactions in a single proof, reducing on-chain footprint
- **Cross-chain verification** — Prove state from other chains without trust assumptions

The network uses **Plonky2/Plonky3** recursive SNARK systems, which provide:

- Fast prover times (sub-second for typical transactions)
- Small proof sizes (~100 bytes)
- Quantum-resistant when paired with lattice-based commitments

### QNT Token

The QNT token serves as the native currency of the Quantus Network with the following utilities:

- **Transaction fees** — All network operations require QNT for gas
- **Staking** — Validators stake QNT to participate in consensus
- **Governance** — Token holders vote on protocol upgrades and parameter changes

### Fee Model

Quantus uses an EIP-1559-style fee mechanism adapted for DAG consensus:

- **Base fee** adjusts dynamically based on network utilization
- **Priority fee** allows users to bid for faster inclusion
- **Base fees are burned**, creating deflationary pressure as usage grows

## Wallet & Developer Experience

### HD Wallet Architecture

The Quantus wallet implements hierarchical deterministic (HD) key derivation using the BIP-32 standard adapted for lattice-based keys:

- Single seed phrase generates unlimited accounts
- Each account has independent post-quantum key pairs
- Hardware wallet support via Ledger and Trezor integrations

### SDK & Tooling

Quantus provides comprehensive developer tooling:

- **quantus-sdk** (TypeScript/JavaScript) — Full-featured SDK for dApp development
- **quantus-cli** — Command-line interface for node operators and developers
- **Block explorer** — Real-time network monitoring with transaction tracing
- **Testnet faucet** — Free testnet QNT for development and testing
